## 1、区分 chatbot、workflow、agent、multi-agent

- **Chatbot**：以对话交互为核心，能说不能做。
- **Workflow**：预设好的固定流程，按图索骥，稳定可预测。
- **Agent**：有脑子的自主执行者，会规划、会用工具、会纠错。
- **Multi-Agent**：一群各怀专长的 Agent 分工协作，解决超复杂问题。

## 2、理解 agent 的基本循环：observe -> think -> act -> observe

Agent 的基本循环可以抽象为 Observe → Think → Act → Observe 的闭环，这是它与 Workflow 最本质的区别。

**Observe（感知）**：Agent 接收环境反馈，包括用户输入、工具返回的结果或外部系统的变化。
**Think（推理规划）**：基于当前观察和历史记忆，大模型进行推理，决定下一步是调用哪个工具、修改计划还是直接给出最终答案。
**Act（行动）**：执行思考结果，比如调用 API、查询数据库或输出给用户。
之后会**立即重新 Observe 行动结果**，形成反馈环路。

这个循环的关键在于**自主性与自我修正**。Agent 不是按预设路径走完就停，而是不断根据中间结果动态调整策略。比如让它‘查北京天气并判断是否要带伞’，它可能先思考需要调用天气工具，行动后观察到返回的温度和降雨概率，再思考‘需不需要推荐带伞’，最后输出结论。

在实际工程实现里，这个循环常体现为 **ReAct 模式**，也就是 Reasoning 和 Acting 交替进行。和 Workflow 固定的 **有向无环图（Directed Acyclic Graph, DAG）**不同，Agent 的路径是实时生成的，有试错能力。这也是为什么 Agent 适合解决开放、路径不确定的复杂任务。”

**这个循环能跑起来，必须有规划、记忆、工具调用三个基础能力支撑，否则就只是空转。**

## 3、明白什么时候不该用 agent：任务可预测、流程稳定、普通脚本能解决时，agent 反而增加不确定性

判断何时不该用Agent的核心就一句话：**能用确定性工程成本解决的问题，绝不引入概率性智能**。具体来说，当任务路径完全可穷举、没有动态决策点，比如标准退款流程，直接用Workflow画DAG跑就行，上Agent反而可能让它跳过关键校验；当结果要求强一致和可复现，比如金融对账，Agent的推理随机性会带来不可解释的风险；当十几行脚本就能完美搞定，比如定时发报表，换成Agent反而是引入上万倍的延迟和幻觉可能。记住，Agent的自主性如果不是完成任务必须付出的代价，就是纯粹的负债，滥用是典型过度工程化。

## 4、[Anthropic: Building effective agents](https://www.anthropic.com/engineering/building-effective-agents) 笔记

这篇文章是Anthropic官方发布的Agent工程实践指南，核心可以用四个字概括：**大道至简**。这篇文章相当于从工业界视角，系统性地印证了Agent领域的关键设计原则。

------

### 📝 文章核心要点总结

- **架构二分法**：将LLM应用明确分为 **Workflow**（代码控制流程）和 **Agent**（模型自主决策流程）。选择取决于任务是否需要模型的动态决策能力。
- **第一性原理**：始终选择最简单的方案，只在必要时增加复杂度。警惕框架带来的抽象层，建议从直接调用LLM API开始。
- **五大核心模式**：文章将其抽象为5种可组合、可复用的模式，构成了Agent系统的“设计模式”：

| 模式                                      | 核心思想                                               | 适用场景                                           |
| :---------------------------------------- | :----------------------------------------------------- | :------------------------------------------------- |
| **1. 提示链 (Prompt Chaining)**           | 将任务分解为顺序执行的步骤，上一步输出作为下一步输入。 | 任务可被清晰地分解为固定子任务。                   |
| **2. 路由 (Routing)**                     | 根据输入类型，将任务分发给专门的处理器。               | 存在多种类别，需要差别化处理。                     |
| **3. 并行化 (Parallelization)**           | 同时执行多个子任务，提高效率或获取多维度视角。         | 任务可被分解为独立的并行子任务，或多视角投票。     |
| **4. 编排-工作者 (Orchestrator-Workers)** | 中央LLM动态拆解任务，分发给“工作者”，最后汇总结果。    | 子任务无法预知，需要动态分配（如复杂的代码修改）。 |
| **5. 评估-优化 (Evaluator-Optimizer)**    | 一个LLM生成，另一个评估并提供反馈，形成迭代优化循环。  | 有明确的评估标准，且迭代能带来可衡量的提升。       |

### 🎯 面试重点考察清单

面试官通过这篇文章，主要想考察你是否具备**架构师级别的判断力**，懂得权衡、选型和规避风险：

1. **核心概念辨析**：能否清晰阐述工作流（Workflow）和智能体（Agent）的根本区别？关键在于**决策权（控制流）在谁手里**：前者是代码，后者是模型。

   **工作流（Workflow）** 是代码通过预设的 **有向无环图（DAG）** 来控制流程，路径确定、可预测、稳定。**Agent** 是大模型自主决策，通过 **观察-思考-行动循环** 来动态生成路径，具备试错和修正能力。

   **选择标准只有一个：任务路径是否可被穷举。**

   **该用工作流时**：任务步骤清晰、边界明确、容错率低。比如标准退款、数据管道、审批流程，用固定DAG就能覆盖所有分支，上Agent反而是引入不可控风险。

   **该用Agent时**：任务路径不固定，需要模型边走边看、动态决策。比如开放式调研、复杂代码调试、多步工具协同解决问题的场景。此时灵活性是刚需，确定性工程无法覆盖所有可能性。

   **一句话总结：能用流程图画清楚的，用工作流；画不清楚、必须边走边想的，才用Agent。**

2. **架构选型与权衡**：这是最核心的考点。面试官会给出具体业务（如客服、代码生成），让你判断用哪种模式或组合实现。你需要能清晰论述 **Latency（延迟）**、**Cost（成本）** 和 **Task Performance（任务性能）** 之间的权衡。

3. **框架的批判性思维**：当被问到用LangChain等框架的经验时，高水平回答应能辩证看待：承认其加速原型开发的价值，同时指出其增加抽象层、引入额外复杂性和调试困难的缺点。

4. **场景应用能力**：你需要能结合文章提到的 **SWE-bench（代码智能体）** 和 **客户支持** 两个场景，准确分析其成功要素，比如可验证的奖励信号和反馈闭环。

5. **设计原则与最佳实践**：对“保持简单”、“透明性”、“精心设计ACI”等原则不能停留在口号，要能结合具体工程案例（如绝对路径vs相对路径）说明如何落地。

6. **“有所不为”的智慧**：这是判断资深与否的分水岭。要能清晰回答“什么时候不该用Agent”，理解在确定性高的场景下，引入Agent的概率性智能是增加了系统脆弱性而非能力。

## 5、[OpenAI: A practical guide to building agents](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/) 笔记

### 📝 文章核心要点总结

与 Anthropic 侧重“大道至简”的工程哲学不同，OpenAI 的这篇文章更像一份**从 0 到 1 的落地施工图**，在强调简单性原则的同时，给了大量可直接使用的决策框架和可执行的实操建议。

核心要点如下：

- **1. 明确“什么是 Agent”**
  文章给出了一个清晰的边界定义：Agent 是一个能**独立完成工作流的系统**，其核心特征在于**用 LLM 来管理和执行工作流**，并能主动纠错或向人类求助。它强调的是**系统化地代表用户办事**，而非仅仅对话。
- **2. Agent 的“三要素”构成**
  这是一个很实用的简化模型，任何 Agent 在本质上都由三个部分组成：
  - **模型**：负责推理与决策的大脑。
  - **工具**：对外部世界采取行动的接口（分为获取数据的工具、执行动作的工具，以及代理本身可作为工具）。
  - **指令**：指导模型行为和设定边界的护栏。
- **3. 何时该用 Agent 的明确信号**
  文章指出了三类适合 Agent 的场景，可作为我们面试时的判断依据：
  - **复杂决策**：涉及细微判断和上下文感知（如退款审批）。
  - **难以维护的规则系统**：大量复杂规则导致维护成本极高。
  - **严重依赖非结构化数据**：需要深度理解自然语言（如处理保险理赔）。
- **4. 两大编排模式**
  - **单智能体**：**核心推荐路径**。主张先用一个 Agent 通过不断增加工具来应对复杂性，只在遇到瓶颈时才考虑拆分。
  - **多智能体**：分为“经理模式”和“去中心化模式”，仅在单智能体因指令过于复杂或工具选择频繁出错时才引入。
- **5. 安全与防护（Guardrails）**
  文章将安全作为核心组件，强调**分层防御机制**，包括基于 LLM 的检查、基于规则的检查，以及对不同风险等级的工具执行不同级别的管控。同时强调了**人机协同（Human-in-the-Loop）** 作为关键安全网，尤其在处理高风险操作时。
- **6. 实用工具箱**
  提供了可直接借鉴的技巧，如：用最强模型建立性能基线再降级、用提示模板减少重复代码，以及模型选型的成本-延迟权衡。

### 🎯 面试重点考察清单

对于这篇 OpenAI 指南，面试官主要想考察你**落地能力**、**系统设计权衡**以及**安全意识**：

- **核心概念辨析**
  - **OpenAI 如何定义 Agent？** 关键在于掌握“用 LLM 管理并执行整个工作流”这一核心标准，这超越了简单的问答或分类。
  - **构成 Agent 的三要素是什么？** 能脱口而出“模型、工具、指令”，并分别解释其作用。
- **架构选型与权衡**
  - **何时选择单智能体，何时转向多智能体？** 应清晰地回答：**首选单智能体**，多智能体只是解决特定瓶颈（如指令冲突、工具混淆）的优化手段，而非默认架构。
  - **“经理模式”与“去中心化模式”的区别？** 能在白板上画出两者的核心区别：控制权是中心化协调，还是在智能体间直接交接。
- **落地与迭代策略**
  - **如何开始第一个Agent项目？** 展示一个务实的思路：先用最强大的模型跑通并建立性能基线，确保效果达标，然后再考虑用更小、更便宜的模型去优化成本和延迟。
  - **如何管理复杂性？** 提及使用提示模板来动态处理不同用例，避免为每个场景维护一套独立的指令，这是非常加分的工程细节。
- **安全与护栏（Guardrails）**
  - **如何设计一个安全可靠的Agent？** 面试官希望听到“分层防御”和“纵深防御”的思路，即组合使用多重护栏，而非依赖单一机制。
  - **如何处理高风险操作？** 必须提到**人机协同（Human-in-the-Loop）**，并列举出自动触发人工介入的明确条件（如超过失败重试阈值、涉及资金或不可逆操作）。
- **对比分析（考察你的视野）**
  - **OpenAI 与 Anthropic 指南有何异同？** 高水平的回答会指出：两者都推崇“简单至上”，但 Anthropic 侧重**不可变的基础组件与架构模式**，而 OpenAI 提供了一个**更具象化的渐进式落地框架**，并极度强调**安全护栏**和**人机协同**。
